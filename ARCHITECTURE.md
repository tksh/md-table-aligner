# ARCHITECTURE.md — MD Table Aligner 技術詳細

## 全体構成

本プロジェクトはビルドステップや外部ライブラリ・フレームワークを持たない、単一の `index.html` で完結する静的なクライアントサイドアプリケーションです。サーバーサイドの処理は一切なく、フォント読み込み・解析・テーブル整形のすべてがブラウザ内の JavaScript で完結します。

## 依存する Web API

- **Local Font Access API**（`window.queryLocalFonts()`）— OS にインストールされたフォントの列挙、および `blob()` によるフォント実体（SFNT バイナリ）の取得
- **FontFace API**（`new FontFace()`, `document.fonts.add()`）— 取得したフォントデータをページ内で実際に使用可能にする
- **Canvas 2D API**（`CanvasRenderingContext2D.measureText()`）— 等幅判定・日英幅比・テーブル列幅計算、すべての幅測定の基盤
- **Clipboard API**（`navigator.clipboard.writeText()`）— 整形結果のコピー

これら以外の外部依存はありません。

## フォント読み込みフロー

1. ユーザーの明示的なクリック（API 仕様上、ユーザーアクティベーションが必須）により `loadFonts()` を実行
2. `window.queryLocalFonts()` を呼び出し、システムの全フォント（`FontData[]`）を取得
3. 各 `FontData` について `blob()` で SFNT バイナリを取得し、`FontFace` として `document.fonts` に登録・ロード。同時に blob を保持し、後述の CJK 判定に使用
   - 現在は `for...of` で逐次 `await` しているため、フォント数が非常に多い環境では読み込みに時間がかかる可能性がある（将来的に `Promise.all` で並列化する余地あり）
4. `family` 名の重複を除いたユニークなリストに対して、等幅判定・CJK 判定・幅比計算を実行
5. 結果を `<select>` に `<optgroup>` で分類して反映（CJK対応 / CJK拡張専用 / ASCII のみ / プロポーショナル）

## 等幅判定アルゴリズム

OpenType の `post` テーブルが持つ `isFixedPitch` フラグには依存せず、**実際の描画幅を Canvas で測定して判定**しています。

理由: 日英幅比をカスタマイズしたプログラミング用フォントでは、内部フラグと実際のグリフ幅が必ずしも一致しないケースがあるため、メタデータより実測のほうが信頼できると判断しました。

```js
function isMonospaceAlphabet(fontFamily, size) {
  const testChars = ['i', 'l', 'W', 'M', '0', '.', 'x', 'A'];
  const widths = testChars.map(ch => ctx.measureText(ch).width);
  const first = widths[0];
  const tolerance = 0.05; // px
  return widths.every(w => Math.abs(w - first) < tolerance);
}
```

幅の異なりやすい文字（`i`/`l` の細い文字と `W`/`M` の太い文字）を混在させることで、似た文字同士の比較による誤判定を避けています。`tolerance` はレンダリング環境によるサブピクセルの誤差を吸収するための経験的な値です。

## 日英幅比の算出

CJK対応の等幅フォントについてのみ、ASCII 基準文字 `M` と全角基準文字 `あ` の実測幅から比率を算出します。

```js
ratio = ctx.measureText('あ').width / ctx.measureText('M').width;
```

- `ratio <= 1.05` の場合はラベルを表示しない（ASCII 専用の等幅フォントに日英幅比ラベルを冗長表示しないため）
- それ以外の場合、`approximateFraction()` により分母 1〜20 の範囲で最も誤差の小さい分数（例: `3:5`）に丸めてラベル化する

`approximateFraction()` は連分数展開ではなく、**分母を 1 から 20 まで総当たりし、`ratio` との誤差が最小になる分数を選ぶ単純な力任せ探索**です。候補数が少なく計算コストも無視できるため、この方式で十分な精度が得られます。

## CJK フォント判定

等幅と判定されたフォントについて、CJK（中国語・日本語・韓国語）文字の有無を判定します。

### 判定ロジック

1. **OS/2 テーブルの確認**（最優先）
   - `ulUnicodeRange` のビット 27（CJK Unified Ideographs）、ビット 28（CJK Extension A）を確認
   - `ulCodePageRange` のビット 17〜21（日本語 932、簡体字中国語 936、韓国語 949、繁体字中国語 950）を確認
   - OS/2 が CJK を示さない場合、即座に `none` を返す（高速パス）

2. **cmap テーブルの確認**（OS/2 が CJK を示した場合のみ）
   - Format 4 サブテーブル: セグメント範囲が BMP CJK（U+4E00〜U+9FFF）を含むか確認
   - Format 12 サブテーブル: グループ範囲が BMP CJK または CJK 拡張（U+20000 以降）を含むか確認

### 判定結果

- `cjk`: BMP CJK 文字（U+4E00〜U+9FFF）を含む → 「等幅（CJK対応）」カテゴリ
- `cjk_ext`: BMP は持たないが U+20000 以降（ExtB〜ExtG+）を含む → 「等幅（CJK拡張専用）」カテゴリ
- `none`: CJK 文字を含まない → 「等幅（ASCII のみ）」カテゴリ

### TTC (TrueType Collection) 対応

MS Gothic、SimSun 等の Windows 標準フォントは TTC 形式で配布されています。`ttcf` シグネチャを検出し、含まれる各フォントについて個別に判定を行います。

## Markdown/Djot テーブルの解析・整形ロジック

本ツールは CommonMark や Djot の完全な構文解析器ではなく、**パイプ文字ベースの簡易的なジオメトリ解析**のみを行います。Markdown と Djot は基本的なパイプテーブル記法（セルを `|` で区切り、ヘッダー直後に `---` または `:--` / `--:` / `:--:` の区切り行を置く）を共有しているため、この単純な解析だけで両方の記法に自然に対応できます。

### 処理手順

1. 入力テキストを行ごとに走査し、`|` を含み空行でない行を「テーブル行候補」として抽出する
2. 各行を `|` で分割し、先頭・末尾の空セル（行頭行末の `|` に由来するもの）を除去してセル配列を作る
3. 全セルが `/^:?-+:?$/` にマッチする行を「区切り行」として検出する
4. 区切り行の各セルにおける `:` の位置から、列ごとの寄せ（`left` / `center` / `right` / `none`）を判定する
5. 区切り行を除く全行を対象に、列ごとに `measureText()` で測った最大幅を「その列の目標幅」とする（最低でも `---` 3 文字分の幅は確保する）
6. 目標幅と実際のセル内容幅の差分をスペース 1 文字分の幅で割り、必要なスペース数を算出してパディングする（寄せ指定に応じて左・右・両側に配分）
7. 区切り行はダッシュ 1 文字の幅から必要なダッシュ数を逆算し、コロン分の幅を差し引いて再構成する
8. 元の行番号を保持したまま該当行だけを書き換え、非テーブル行はそのまま残す

### 既知の制約（設計上のトレードオフ）

- **簡易パーサーであることの限界**: インラインコード内の `` `a|b` `` やエスケープされた `\|` など、テーブル以外の文脈で現れる `|` を区別できません。個人利用でパイプテーブル以外に `|` を含む行がない前提であれば問題になりませんが、汎用パーサーとしては不完全です。
- **整形は見た目上の調整のみ**: 実測幅にもとづくパディングは、エディタ上やこのツールのプレビューで視覚的に列を揃えるためのものであり、Markdown/Djot のレンダラーにとっては意味を持ちません（多くのレンダラーはセル内の余分な空白を無視して描画するため）。あくまでソースコードとしての可読性向上が目的です。
- **ヒューリスティックな定数**: 等幅判定の許容誤差（0.05px）や日英幅比ラベルの閾値（1.05）は経験的に決めた値です。特殊なフォント・レンダリング環境では調整が必要になる可能性があります。

## 実行環境の制約

- Local Font Access API は secure context 上のトップレベルドキュメントでのみ動作するため、`file://` として直接開くか、自前で http/https 配信する必要があります。
- クロスオリジンの iframe に埋め込まれた場合（Claude.ai の Artifacts プレビューなど）、Permissions Policy のデフォルト許可リストが `self` のみであるため、ブラウザの許可プロンプトが表示される前の段階で `SecurityError` により拒否されます。この挙動は、サイトごとのブラウザ設定（例: Edge の「フォント」権限）とは独立した、より手前の段階でのブロックです。
- モバイル OS（iOS/Android）では API 自体が提供されていません。
